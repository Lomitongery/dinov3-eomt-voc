# EoMT (Encoder-only Mask Transformer) 在 PASCAL VOC 2012 上的实验报告

## 1. 实验配置

### 1.1 模型架构

| 组件 | 配置 |
|------|------|
| Backbone | DINOv3 ViT-Small (`vit_small_patch16_dinov3.lvd1689m`), 23.3M 参数 |
| Decoder | EoMT query-based mask decoder |
| 损失函数 | Mask2Former 匈牙利匹配 + BCE mask (×5.0) + Dice (×5.0) + CE class (×2.0) |
| Query 数量 | 20 (从 100 下调，诊断出 query collapse 问题后调整) |
| 预训练权重 | DINOv3 ImageNet-22K 预训练 |

### 1.2 数据集

| 项目 | 数值 |
|------|------|
| 数据集 | PASCAL VOC 2012 |
| 训练集 | 1464 张图片 |
| 验证集 | 1449 张图片 |
| 类别数 | 21 (20 个物体类 + 1 个背景类) |
| 输入尺寸 | 512 × 512 |

### 1.3 训练超参数

| 参数 | 数值 |
|------|------|
| Batch size | 2 |
| 总轮数 | 50 |
| 每轮步数 | 732 |
| 优化器 | AdamW |
| 基础学习率 | 1 × 10⁻⁴ |
| Layer-wise LR Decay (LLRD) | 0.8 |
| 混合精度 | FP16 |
| GPU | NVIDIA RTX 5070 Laptop (8GB) |

### 1.4 数据增强

训练阶段采用以下数据增强策略：

- ColorJitter: 随机调整亮度、对比度、饱和度和色相
- RandomFlip: 随机水平翻转
- ScaleJitter(0.1, 2.0): 随机缩放，范围 0.1 到 2.0 倍
- Pad: 填充至统一尺寸
- RandomCrop: 随机裁剪至 512 × 512

### 1.5 Mask Annealing 策略

采用多项式衰减 (power=0.9)，4 个 block 交错排布。每个 block 的 mask 点数从起始值线性衰减到终止值：

| Block | 起始点数 | 终止点数 |
|-------|----------|----------|
| Block 0 | 0 | 8784 |
| Block 1 | 5856 | 14640 |
| Block 2 | 10248 | 20496 |
| Block 3 | 14640 | 26352 |

## 2. 训练过程

### 2.1 损失曲线

训练过程中损失函数从初始值 62.0 下降到最终值 39.2。损失下降主要集中在前 10 个 epoch，之后进入缓慢下降阶段。损失下降幅度约为 37%，但这一下降并未有效转化为分割精度的提升。

### 2.2 mIoU 曲线

mIoU 在整个训练过程中几乎没有实质性变化：

- Epoch 0: 3.5%
- Epoch 42 (最佳): 4.3%
- Epoch 49: 4.3%

随机基线 (1/21) 为 4.8%，意味着模型在所有 epoch 上的表现均低于随机猜测。这是一个明确的信号，表明模型未能学习到任何有意义的语义分割能力。

### 2.3 训练时间

- 每轮耗时: 约 1.2 分钟
- 总训练时间: 约 1 小时 (50 轮)
- 对比 PSPNet: 约 30 分钟 (相同 epoch 数)

EoMT 的训练时间约为 PSPNet 的两倍，这主要是因为匈牙利匹配和 mask 头部的计算开销更大。

## 3. 结果分析

### 3.1 Query Collapse 现象

这是实验中最核心的问题。在第一次尝试 (num_q=100) 中，99/100 个 query 预测为 "no_object"，mIoU 仅为 4.0%。将 query 数量下调至 20 后，情况略有改善：

- 18/20 个 query 仍预测为 "no_object"
- no_object 的平均概率从 0.838 下降到 0.522
- 有 8 个 query 处于 "活跃" 状态 (但活跃 query 的预测质量仍然很低)

Query collapse 的本质是：匈牙利匹配算法在训练过程中发现，将 query 分配给 "no_object" 比冒险预测一个错误类别能获得更低的损失。这是一个局部最优陷阱，模型一旦陷入就很难逃脱。

### 3.2 类别预测分析

从逐像素预测的角度分析：

- 模型仅能预测 2-4 个类别
- 背景类占比高达 82.5%
- 其余少数类别 (如 person、car) 偶尔被预测，但边界极其模糊

这表明模型没有学会区分不同语义类别，而是倾向于将所有像素归类为背景。

### 3.3 损失分量分析 (Epoch 50)

| 损失分量 | 数值 | 说明 |
|----------|------|------|
| Mask BCE | 0.34 | 相对较低，mask 预测尚可 |
| Dice | 0.43 | 中等水平 |
| CE class | 1.86 | 远高于其他分量 |

CE class 损失高达 1.86，是 mask 损失的 5 倍以上。这直接说明模型无法学习正确的类别分配。匈牙利匹配将 query 与 GT 实例配对后，模型在分类头上无法稳定地预测对应类别，导致分类损失居高不下。

## 4. 与 PSPNet 对比

### 4.1 性能对比

| 指标 | PSPNet | EoMT |
|------|--------|------|
| Backbone | DINOv3 ViT-S (冻结) | DINOv3 ViT-S (微调) |
| Decoder | PSPNet PPM | EoMT queries + mask head |
| 损失函数 | CrossEntropy (逐像素) | 匈牙利匹配 + BCE + Dice + CE |
| mIoU | ~88% | 4.3% |
| 像素准确率 | ~97% | — |
| 训练时间 | ~30 分钟 | ~60 分钟 |
| 轮数 | 50 | 50 |

### 4.2 关键差异分析

PSPNet 和 EoMT 使用完全相同的 backbone (DINOv3 ViT-S)，但结果差距巨大。核心差异在于解码器设计：

1. **PSPNet 使用逐像素分类**：PPM 模块聚合多尺度上下文信息后，每个像素独立预测类别。262,144 个像素 (512×512) 每个都参与损失计算，梯度信号极其密集。

2. **EoMT 使用 query-based 匹配**：20 个 query 通过匈牙利匹配与 3-5 个 GT 实例配对。每张图片只有 3-5 对匹配参与损失计算，梯度信号极其稀疏。

3. **PSPNet 冻结 backbone**：PSPNet 实验冻结了 DINOv3 backbone，仅训练 PPM 和解码头。这反而成了一个优势——预训练特征被完整保留，PPM 只需学习如何聚合这些特征。

4. **EoMT 微调 backbone**：EoMT 对整个网络进行微调，理论上更灵活，但实际上由于梯度信号稀疏，微调反而破坏了 backbone 的预训练特征，却没有学到有效的分割能力。

## 5. 失败原因剖析

经过系统分析，我们识别出三个相互关联的失败模式：

### 5.1 小数据集上的匈牙利匹配失效

PASCAL VOC 2012 训练集仅有 1464 张图片，每张图片平均包含约 3 个物体实例。整个训练集总共约 4400 个训练实例。20 个 query 需要在 50 个 epoch 内学会稳定的类别分配，但训练实例太少，query 无法收敛到有意义的语义概念。

匈牙利匹配的贪婪性质加剧了这一问题：早期 epoch 中匹配错误的 query 会获得错误的梯度信号，后续 epoch 难以纠正。在小型数据集上，这种错误累积效应尤为明显。

### 5.2 类别不平衡的放大效应

PASCAL VOC 2012 本身存在严重的类别不平衡：

- 背景类出现在 100% 的图片中
- Person、car、chair 等常见类出现在 30-60% 的图片中
- Boat、tv/monitor、bottle 等稀有类出现在不到 5% 的图片中

匈牙利匹配为每个 GT 实例分配恰好一个 query。这意味着稀有类在整个训练过程中只被分配到几十次匹配。相比之下，逐像素方法中每个稀有类像素仍然参与损失计算，只是权重较低。匈牙利匹配将类别不平衡问题从 "像素级加权" 放大为 "实例级稀缺"。

### 5.3 梯度信号稀疏性

这是最根本的问题。两种方法的梯度信号密度对比：

- **逐像素 CE 损失**：每张图片监督 262,144 个像素 (512 × 512)，每个像素提供梯度信号
- **匈牙利匹配损失**：每张图片监督 3-5 个匹配对 (query ↔ GT instance)，每个匹配对提供梯度信号

梯度信号密度相差约 50,000 倍。这意味着 EoMT 的优化景观本质上更加困难——模型需要在极其稀疏的监督信号下，从 20 个 query 中学会 21 个类别的语义分割。

## 6. 结论与建议

### 6.1 核心结论

1. **EoMT 的 query-based 架构不适合 PASCAL VOC 2012 的数据规模**。1464 张训练图片、21 个类别、每张 1-5 个物体实例的数据集无法支撑 query-based 方法的训练。

2. **EoMT 论文报告的成功实验均在更大数据集上完成**：Cityscapes (107 epoch, 5K 图片) 和 ADE20K (31 epoch, 20K 图片)。这暗示 query-based 分割方法存在一个最低数据量门槛，大约在 5K 图片量级。

3. **对于 VOC 规模的数据集，逐像素方法仍然显著优于 query-based 方法**。PSPNet 在相同 backbone 和相同 epoch 数下达到 88% mIoU，而 EoMT 仅为 4.3%。

4. **本实验的负面结果具有科学价值**。它建立了一个 query-based 分割方法可行性的下界——当训练数据少于 5K 图片时，匈牙利匹配 + query 的设计无法有效学习。这一发现对社区选择分割架构具有参考意义。

### 6.2 改进建议

1. **如果继续推进 EoMT 路线**：切换到 Cityscapes 或 ADE20K 数据集。这两个数据集的图片数量和实例多样性足以支撑 query-based 方法的训练。

2. **如果坚持使用 VOC 数据集**：使用逐像素解码器。PSPNet 已经证明在 VOC 上可以达到 88% mIoU，DeepLab 系列也是成熟的选择。

3. **可能的改进方案**：在 COCO (118K 图片) 上预训练 EoMT，然后在 VOC 上微调。大规模预训练应该能缓解数据稀缺问题，让 query 在预训练阶段就学会稳定的语义概念，微调时只需适应 VOC 的类别分布。

4. **架构层面的改进**：可以考虑混合方案——在早期阶段使用逐像素损失辅助训练 query，待 query 稳定后再切换到纯匈牙利匹配。这类似于课程学习 (curriculum learning) 的思路。
